# VEDA Multi-Sensor TCP Control System 실행 과정

본 문서는 VEDA Multi-Sensor TCP Control System의 빌드, 배포, 실행 및 각 기능별 시연 과정을 단계별로 정리한 가이드입니다.

---

## 1. 실행 전 준비

### 대상 환경
* **서버 장치**: Raspberry Pi 4 (AArch64)
* **클라이언트**: Ubuntu Linux 또는 WSL Ubuntu (x86_64)
* **서버 포트**: TCP `8000`
* **배포 경로**: `<DIR>>`

### 필요 도구
* `make`
* `gcc`
* `aarch64-linux-gnu-gcc` (크로스 컴파일러)
* **Raspberry Pi sysroot**: `~/rpi-sysroot` (wiringPi 헤더 및 라이브러리가 내부에 포함되어 있어야 함)

### 하드웨어 연결 (핀맵)
| 장치 | wPi Pin | BCM GPIO | Physical Pin | 비고 |
|------|---------|----------|--------------|------|

## 3. 빌드

빌드 시 기존의 크로스 컴파일용 `Makefile`을 삭제하고, `Makefile.simple`을 `Makefile`로 대체하여 사용해야 합니다.

프로젝트 루트에서 다음 명령을 실행하여 기존 `Makefile`을 삭제하고 `Makefile.simple`을 복사한 뒤 빌드합니다.

```bash
rm -f Makefile
cp Makefile.simple Makefile
make
```

### 정상 빌드 결과물
* `server`: Raspberry Pi에서 실행할 TCP 웹 서버
* `client`: 호스트 PC에서 실행할 CLI 클라이언트
* `libled.so`: LED 제어 드라이버
* `libbuzzor.so`: 부저 제어 드라이버
* `libphotoresistor.so`: 조도센서 드라이버
* `libsegment.so`: 7세그먼트 드라이버

### 빌드 초기화 후 다시 빌드할 때
```bash
make clean && make
```

> [!TIP]
> **빌드 실패 시 확인할 항목**
> * `aarch64-linux-gnu-gcc` 크로스 컴파일러 설치 여부
> * `~/rpi-sysroot` 디렉토리와 내부에 wiringPi 헤더 및 라이브러리 존재 여부
> * `Makefile` 내부의 `SYSROOT` 경로 설정 확인

## 4. Raspberry Pi로 배포

배포를 위해 `Makefile` 내에 정의된 `deploy` 타겟을 활용하여 한 번에 빌드 및 업로드를 자동화하거나, 수동으로 전송할 수 있습니다.

### 방법 A: Make 명령어로 자동 배포 (권장)
```bash
# 기본값(RPI_USR=sunbi, RPI_IP=192.168.0.100, RPI_DIR=/home/sunbi/prj)으로 배포할 때
make deploy

# 특정 IP, 계정, 경로를 지정하여 배포할 때
make deploy RPI_IP=192.168.1.50 RPI_USR=pi RPI_DIR=/home/pi/myproject
```

### 방법 B: 수동 명령어로 배포
```bash
# 1. Raspberry Pi에 배포 폴더 생성
ssh <USR>@<RPI_IP> "mkdir -p <DIR>"

# 2. 서버 실행 파일 및 동적 라이브러리(.so) 전송
scp server lib*.so <USR>@<RPI_IP>:<DIR>

# 3. 웹 리소스 디렉토리 전송
scp -r resources <USR>@<RPI_IP>:<DIR>

# 4. 배포 완료 확인
ssh <USR>@<RPI_IP>
cd <DIR>
ls -F
```

**확인 대상**: `server*`, `libled.so`, `libbuzzor.so`, `libphotoresistor.so`, `libsegment.so`, `resources/`

## 5. 서버 실행

Raspberry Pi에서 서버를 구동합니다.

```bash
cd <DIR>
export LD_LIBRARY_PATH=.:$LD_LIBRARY_PATH
```

### 디버그용 전경(Foreground) 실행
```bash
./server -f
```
**정상 실행 로그 예시**:
```text
GPIO initialized successfully
Server started on port 8000 with poll/multi-thread...
```

### 백그라운드 상시 구동 (데몬 모드)
```bash
./server
```
* 데몬 모드로 실행 시 터미널 프롬프트가 바로 반환되며, 백그라운드에서 동작합니다.

### 서버 종료 방법
* **전경 모드**: `Ctrl + C` 입력 시 `sigaction()` 핸들러가 `server_running` 플래그를 `0`으로 만들고 안전하게 종료됩니다.
* **데몬 모드**: `sudo killall server` 또는 `sudo kill <PID>` 명령을 실행하여 프로세스를 종료합니다.

## 6. 웹 대시보드 실행

호스트 PC의 브라우저에서 아래 주소로 접속합니다.

```text
http://<RPI_IP>:8000/
```

메인 페이지에서 **LED Control**, **7-Segment**, **Buzzer Scale**, **LDR Sensor** 4개의 메뉴 카드를 확인할 수 있습니다.

## 7. LED 제어 시연

### 웹 대시보드 시연
1. `http://<RPI_IP>:8000/led.html` 접속
2. **ON** 버튼 클릭 $\rightarrow$ LED가 최대 밝기로 점등되는지 확인
3. **MID** 버튼 클릭 $\rightarrow$ LED가 중간 밝기로 감소하는지 확인
4. **밝기 슬라이더** 조절 $\rightarrow$ PWM(0~1024) 값에 따라 밝기가 미세하게 변하는지 확인
5. **OFF** 버튼 클릭 $\rightarrow$ LED가 소등되는지 확인

### API 직접 호출 검증
```bash
curl http://<RPI_IP>:8000/led/on
curl http://<RPI_IP>:8000/led/mid
curl "http://<RPI_IP>:8000/led/brightness?value=300"
curl http://<RPI_IP>:8000/led/off
```
* **기대 응답**: `OK` (HTTP 상태 코드 `200 OK`)
* **잘못된 인자 입력 시**: `HTTP/1.1 400 Bad Request` 응답 확인

## 8. 부저 제어 시연

### 웹 대시보드 시연
1. `http://<RPI_IP>:8000/buzzer.html` 접속
2. **do, re, mi** 등 건반 버튼 클릭 $\rightarrow$ 부저가 짧게 울리는지 확인
3. **FEIN ALERT** 버튼 클릭 $\rightarrow$ FEIN 스타일 알림 멜로디가 연속 재생되는지 확인
4. **OFF** 버튼 클릭 $\rightarrow$ 즉시 소리가 멈추는지 확인

### API 직접 호출 검증
```bash
curl http://<RPI_IP>:8000/buzz/do
curl http://<RPI_IP>:8000/buzz/5
curl http://<RPI_IP>:8000/buzz/880
curl http://<RPI_IP>:8000/buzz/fein
curl http://<RPI_IP>:8000/buzz/off
```

---

## 9. 조도센서 시연

### 웹 대시보드 시연
1. `http://<RPI_IP>:8000/pr.html` 접속
2. **아날로그 ADC 값**(`0~255`)과 **디지털 값**(`0` 또는 `1`)이 실시간 표시되는지 확인
3. 조도센서를 손으로 가려 어둡게 하거나, 불빛을 비춰 밝게 하여 데이터가 실시간으로 변하는지 확인
4. **Enable Server Auto Control** 버튼 클릭 $\rightarrow$ 자동 조도 제어 스레드 구동 시작
5. 센서 주변이 어두워지면(디지털 값 `0`) LED가 자동으로 켜지고, 밝아지면(디지털 값 `1`) LED가 꺼지는지 확인
6. **Disable Server Auto Control** 버튼 클릭 $\rightarrow$ 자동 조도 제어 중지

### API 직접 호출 검증
```bash
curl http://<RPI_IP>:8000/pr
curl http://<RPI_IP>:8000/pr/digital
curl http://<RPI_IP>:8000/pr/auto/start
curl http://<RPI_IP>:8000/pr/auto/stop
```

## 10. 7세그먼트 시연

### 웹 대시보드 시연
1. `http://<RPI_IP>:8000/segment.html` 접속
2. **Start (Default 9-0)** 버튼 클릭
3. 세그먼트에 `9`부터 `0`까지 1초 간격으로 감소하며 표시되는지 확인
4. `0` 도달 시 LED가 켜지고 부저 알림음이 함께 발생하는지 확인
5. 카운트다운 도중 **OFF** 버튼 클릭 $\rightarrow$ 세그먼트와 LED가 즉시 꺼지고 카운트다운이 취소되는지 확인
6. **Countdown Start Value** 입력 칸에 숫자(예: `5`)를 입력하고 **Start Custom** 클릭 $\rightarrow$ 지정한 숫자부터 카운트다운 동작 확인
7. **Direct Display**에서 숫자를 클릭하여 단발성으로 해당 숫자가 즉시 유지 표시되는지 확인

### API 직접 호출 검증
```bash
curl http://<RPI_IP>:8000/segment/start
curl http://<RPI_IP>:8000/segment/5
curl http://<RPI_IP>:8000/segment/show/8
curl http://<RPI_IP>:8000/segment/off
```

## 11. CLI 클라이언트 시연

호스트 PC 터미널에서 다음 명령어로 서버에 접속합니다.

```bash
./client <RPI_IP>
```

### 실행 UI 및 메뉴 작동
접속에 성공하면 아래 메뉴가 루프 형태로 출력됩니다.
```text
1 LED  2 Buzzer  3 7-Segment  4 Photoresistor  5 Quit
Select menu:
```

* **1 LED**: `ON`, `MID`, `OFF`, `0~1024` 범위의 숫자 입력 검증. 범위 이외 입력 시 에러를 검증하고 복귀
* **2 Buzzer**: `do`, `re`, `mi`, `fa`, `sol`, `la`, `si`, `do2`, `off` 등 입력 시 재생
* **3 7-Segment**: `start`, `off`, `0~9`(1자리), `show/0~9`(1자리) 입력. 비정상 입력(예: `abc`, `00`, `show/10`) 시 `Invalid segment command!` 경고 후 복귀
* **4 Photoresistor**: 아날로그/디지털 센서 조회, 서버 자동 LED 제어 ON/OFF 서브 메뉴 진입
* **5 Quit**: 자동 제어 중지 요청을 서버로 보내고, 소켓을 닫은 뒤 안전하게 종료

### 추가 검증 시나리오
* **Ctrl+C 종료 안정성**: CLI 클라이언트 실행 중에 `Ctrl+C`를 입력하면 `sigaction` 핸들러가 플래그를 받아 메인 스레드에서 `pr/auto/stop`을 안전하게 호출하고 소켓을 클린업한 뒤 안전 종료됩니다.
* **연결 실패 지수 백오프**: 서버가 작동하지 않는 상황에서 클라이언트를 실행하거나 명령을 보내면 최대 3회까지 시간 간격을 늘려가며(1초 $\rightarrow$ 2초 $\rightarrow$ 4초) 지수 백오프로 연결 재시도를 수행하고, 최종 실패 시 메뉴로 안전하게 복귀합니다.

## 12. 스레드 모니터링 시연

### 웹 대시보드 실시간 관찰
1. `http://<RPI_IP>:8000/` 메인 대시보드 페이지에 접속합니다.
2. 페이지 하단의 **Active Thread Monitor** 테이블을 확인합니다. (초기 상태: *No active threads running*)
3. 별도의 터미널 창을 열거나 다른 탭으로 7세그먼트 카운트다운을 동작시킵니다. (`GET /segment/start`)
4. 실시간으로 테이블에 카운트다운 처리를 담당하는 쓰레드 정보(TID, Client IP, Active Task: `segment/start`, Uptime)가 1.5초 주기로 노출되는지 확인합니다.
5. 카운트다운이 종료되어 0이 되는 순간, 또는 **OFF** 버튼으로 강제 종료하는 순간 목록에서 해당 쓰레드가 자동으로 삭제되는지 확인합니다.

### API 직접 검증
```bash
curl http://<RPI_IP>:8000/api/threads
```
* **활성 쓰레드가 없을 때**: `[]`
* **카운트다운 중일 때**: `[{"tid":12345678,"ip":"192.168.x.x","cmd":"segment/start","uptime":4}]`
