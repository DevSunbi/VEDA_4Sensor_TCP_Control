# 🎛️ VEDA 4-Sensor TCP Control

Raspberry Pi 4에서 동작하는 **TCP 기반 IoT 장치 원격 제어 시스템**입니다.  
웹 브라우저를 통해 LED, 부저, 7세그먼트, 조도센서를 실시간으로 제어하고 모니터링할 수 있습니다.

## 🔧 빌드 환경 및 빌드 방법

본 시스템은 **라즈베리 파이 타겟용 크로스 컴파일**과 **호스트 PC용 네이티브 컴파일**을 혼합하여 빌드 성능과 편의성을 극대화했습니다.

### 1. 사전 요구사항

| 구분 | 컴파일러 / 환경 | 대상 바이너리 | 비고 |
|------|----------|--------------|------|
| **호스트 PC (Ubuntu)** | `gcc` (x86_64 네이티브) | `client` | 호스트 환경에서 즉시 실행 가능 |
| **타겟 보드 (RPi4)** | `aarch64-linux-gnu-gcc` | `server`, `lib*.so` | 라즈베리 파이용 크로스 컴파일 |
| **타겟 Sysroot** | `~/rpi-sysroot` | - | Raspberry Pi OS 파일시스템 복사본 필요 |
| **라이브러리** | `wiringPi` | - | 타겟 보드 및 크로스 빌드 환경 내 링크 필요 |

* **크로스 컴파일러 설치 (Ubuntu Host):**
  ```bash
  sudo apt update
  sudo apt install gcc-aarch64-linux-gnu
  ```

### 2. 빌드 실행

호스트 PC의 프로젝트 루트(`~/src`)에서 아래 명령어를 실행합니다:

* **전체 빌드 (서버 크로스 컴파일 + 클라이언트 네이티브 컴파일):**
  ```bash
  make
  ```
* **클린 및 재빌드:**
  ```bash
  make clean && make
  ```

빌드 결과물:
* `server`: 라즈베리 파이용 크로스 컴파일 실행 파일
* `client`: 호스트 PC(Ubuntu)용 네이티브 컴파일 실행 파일
* `libled.so`, `libbuzzor.so`, `libsegment.so`, `libphotoresistor.so`: 라즈베리 파이용 크로스 컴파일 공유 라이브러리

## 🚀 배포 및 실행

### 1. 파일 전송 (호스트 → Raspberry Pi)

크로스 컴파일된 서버 및 공유 라이브러리와 웹 프론트엔드 HTML을 라즈베리 파이로 전송합니다.

```bash
# 실행 파일 및 공유 라이브러리 전송
scp server lib*.so sunbi@<RPI_IP>:/home/sunbi/projectHome/

# HTML 프론트엔드 파일 전송
scp src/*.html sunbi@<RPI_IP>:/home/sunbi/projectHome/src/
```

### 2. 서버 실행 (Raspberry Pi)

라즈베리 파이에 접속하여 서버를 구동합니다:

```bash
cd /home/sunbi/projectHome
export LD_LIBRARY_PATH=.:$LD_LIBRARY_PATH
./server
```

### 3. 웹 대시보드 접속

웹 브라우저를 열고 다음 주소로 접속합니다:
```
http://<RPI_IP>:8000/
```

### 4. 대화형 콘솔 클라이언트 실행 (호스트 PC에서 원격 실행)

클라이언트는 호스트 PC 네이티브 바이너리로 빌드되어 있으므로, 라즈베리 파이에 접속할 필요 없이 **호스트 PC의 터미널에서 즉시 실행**하여 원격 제어할 수 있습니다.

```bash
./client <RPI_IP>
```

**제공 메뉴 및 기능:**
* **1: LED 제어** (ON/MID/OFF 및 0~1024 사이의 PWM 듀티 입력 제어)
* **2: 부저 음계 제어** (도, 레, 미... 등 재생 및 정지)
* **3: 7세그먼트 카운트다운 제어** (지정된 숫자 입력 시 카운트다운 후 부저 알림)
* **4: 조도 센서 모니터링** (실시간 아날로그 조도 및 디지털 상태 조회)
* **5: 프로그램 종료** (안전한 소켓 연결 해제 및 리소스 반환)

## 🔑 핵심 아키텍처 및 설계 포인트

### 1. 웹-서버 이중 루프 충돌 방지 (7세그먼트)
* **문제점:** 이전 구조에서는 웹 프론트엔드 자바스크립트의 `setInterval` 루프와 서버의 7세그먼트 카운트다운 루프가 동시에 돌면서 핀 상태를 중복으로 제어하여 화면이 깜빡이거나 딜레이가 발생하는 충돌이 있었습니다.
* **해결책:** 7세그먼트 제어를 **단발성 출력 API (`/segment/show/<digit>`)**와 **서버 단 독립 카운트다운 루프 (`/segment/<digit>`)**로 이원화하였습니다. 카운트다운 타이머 구동 시에는 프론트엔드가 상태 조회를 멈추거나 갱신 빈도를 동기화하여 물리 핀의 경합을 원천적으로 차단했습니다.

### 2. 진정한 서버 주도 조도 센서 백그라운드 제어 (Auto LDR Mode)
* **문제점:** 기존의 자동 제어는 브라우저가 조도 API를 계속 호출하며 LED API를 때리는 브라우저-클라이언트 의존적 구조였습니다.
* **해결책:** 서버 내부에 독립적인 `auto_pr_control_thread` 백그라운드 스레드를 신설하여 `/pr/auto/start` 요청 시 즉시 동작하고, 조도가 어두워지면(1) LED를 켜고 밝아지면(0) 끄는 임베디드 핵심 로직을 서버 독단적으로 처리하도록 하였습니다.

### 3. 멀티스레드 환경의 GPIO 뮤텍스 최적화
* 카운트다운 루프나 대기 시간(`delay(1000)`) 동안 뮤텍스를 쥐고 있으면 다른 제어 API(예: 부저 끄기, LED 제어)가 뮤텍스 획득을 위해 블로킹 상태가 되어 서버 응답 속도가 현저히 저하됩니다.
* 이를 막기 위해 물리 핀을 실제로 제어하는 극소수의 코드 블록에만 락(`pthread_mutex_lock`)을 걸고, `delay()` 대기 시점에는 락을 바로 해제하여 다른 요청 스레드가 병렬적으로 즉각 수행될 수 있도록 락 정밀도를 최적화했습니다.

### 4. 클라이언트 강인성 및 예외 처리 (Signal Handling)
* 클라이언트 프로그램(`client.c`)의 안정적 종료를 위해 `SIGINT` (Ctrl+C) 수신 시 열려 있는 서버 소켓 파일 디스크립터(`sockfd`)를 명시적으로 `close()`하고 메모리를 정리한 후 정상 종료하도록 시그널 핸들러를 구축했습니다.
* 사용자가 조작 중 실수로 누를 수 있는 `SIGTSTP` (Ctrl+Z, 일시중지) 및 `SIGQUIT` (Ctrl+\, 코어덤프 종료) 시그널은 무시(`SIG_IGN`) 처리하여 예기치 않은 오작동이나 좀비 연결 상태를 방지했습니다.
